# Employee Interview Analyzer - 员工访谈结果智能分析技能

专为互联网公司 HR 设计的**访谈结果智能分析**技能。输入访谈记录/纪要，自动识别访谈场景，智能匹配分析方法论，输出结构化洞察报告。

> **与 skill-b（employee-interview-generator 提纲生成器）的关系：**
> - **skill-b（提纲生成器）** → 访谈**前**：生成专业的访谈提纲
> - **skill-a（本技能/分析器）** → 访谈**后**：分析访谈结果，提取深度洞察
> - 两个技能共享相同的方法论体系，形成完整的**访谈前→访谈后**闭环

### Step 1: 智能场景识别引擎（自动执行） ⭐

分析引擎通过**关键词匹配 + 语义理解 + 上下文推断**三层机制，自动识别访谈场景类型。无需询问用户，直接基于上下文进行识别。

> **核心原则：在开始正式分析前，先根据上下文修正访谈记录中的错别字和语句不通顺的地方，确保分析基于准确清晰的文本。**

#### 修正范围

**1. 错别字修正**
- 常见同音字错误（如：的/得/地、在/再、已/以等）
- 形近字错误
- 输入法选词错误

**2. 语句不通修正**
- 语序混乱的句子
- 缺少主语或宾语的句子
- 重复冗余的表达
- 标点符号错误

**3. 上下文理解修正**
- 基于上下文推断可能的正确用词
- 保留口语化表达，但修正明显错误
- 不改变原意，仅提升可读性

#### 修正规则

| 规则类型 | 示例 | 修正后 |
|---------|------|-------|
| **错别字** | "这个项目做的不错" | "这个项目做得不错" |
| **错别字** | "我已经以经准备好了" | "我已经已经准备好了" |
| **语句不通** | "因为很多问题，所以导致" | "因为有很多问题，所以导致" |
| **语句不通** | "他跟我说了很多，然后，然后..." | "他跟我说了很多，然后..." |
| **标点错误** | "我们要解决问题，然后才能前进" | "我们要解决问题，然后才能前进" |

#### 修正流程

```python
def correct_interview_text(text: str) -> dict:
    """
    修正访谈文本中的错误
    返回: {
        "corrected_text": "修正后的文本",
        "corrections": [
            {
                "original": "原文片段",
                "corrected": "修正后片段",
                "type": "错别字/语句不通/标点错误",
                "reason": "修正理由"
            }
        ]
    }
    """
    corrections = []
    corrected_text = text
    
    # 1. 错别字修正（基于常见错误字典）
    corrected_text, typo_corrections = correct_typos(corrected_text)
    corrections.extend(typo_corrections)
    
    # 2. 语句不通修正（基于语法规则）
    corrected_text, grammar_corrections = correct_grammar(corrected_text)
    corrections.extend(grammar_corrections)
    
    # 3. 标点符号修正
    corrected_text, punctuation_corrections = correct_punctuation(corrected_text)
    corrections.extend(punctuation_corrections)
    
    # 4. 上下文一致性检查
    corrected_text, context_corrections = check_context_consistency(corrected_text)
    corrections.extend(context_corrections)
    
    return {
        "corrected_text": corrected_text,
        "corrections": corrections
    }
```

#### 修正输出

在分析报告中，文本修正部分会以附录形式呈现：

```markdown
## 📝 文本修正记录（附录）

本次分析对访谈文本进行了以下修正：

| 类型 | 原文 | 修正后 | 理由 |
|------|------|-------|------|
| 错别字 | 做的不错 | 做得不错 | 地/得/用法规范 |
| 语句不通 | 因为很多问题，所以导致 | 因为有很多问题，所以导致 | 补充完整句子结构 |

**修正原则：**
- 不改变原意，仅提升可读性
- 保留口语化表达，仅修正明显错误
- 基于上下文推断，确保理解准确
```

#### 注意事项

- ⚠️ **保留原汁原味**：口语化表达不必过度修正，只需确保意思清楚即可
- ⚠️ **避免误判**：不确定的用词不要随意修改，保持原样
- ⚠️ **记录修正**：所有修正都要记录在报告中，方便追溯
- ✅ **提升准确性**：文本修正有助于提升后续方法论匹配的准确性

#### 场景识别算法

```python
def identify_scene(transcript: str) -> dict:
    """
    智能识别访谈场景
    返回: {
        "primary_scene": "离职访谈",       # 主场景
        "confidence": 0.92,                # 置信度
        "secondary_scenes": ["挽留访谈"],  # 次要场景
        "scene_signals": [...],            # 识别依据
        "sub_type": "主动离职"             # 子类型（如有）
    }
    """
    scores = {}
    
    # 第一层：关键词匹配（权重 40%）
    for scene, keywords in SCENE_KEYWORDS.items():
        primary_hits = count_hits(transcript, keywords["primary"])
        secondary_hits = count_hits(transcript, keywords["secondary"])
        scores[scene] = primary_hits * 3 + secondary_hits * 1
    
    # 第二层：语义分析（权重 40%）
    # 分析整体语义场景、情感基调、讨论主题
    semantic_scores = analyze_semantics(transcript)
    for scene, score in semantic_scores.items():
        scores[scene] += score * 4
    
    # 第三层：上下文推断（权重 20%）
    # 基于参与者身份、时间节点、讨论结构推断
    context_scores = infer_from_context(transcript)
    for scene, score in context_scores.items():
        scores[scene] += score * 2
    
    # 归一化并排序
    total = sum(scores.values())
    normalized = {k: v/total for k, v in scores.items()}
    sorted_scenes = sorted(normalized.items(), key=lambda x: -x[1])
    
    return {
        "primary_scene": sorted_scenes[0][0],
        "confidence": sorted_scenes[0][1],
        "secondary_scenes": [s[0] for s in sorted_scenes[1:3] if s[1] > 0.15],
        "scene_signals": extract_signals(transcript, sorted_scenes[0][0])
    }
```

#### 方法论匹配算法

```python
MAX_METHODOLOGIES = 3  # ⭐⭐⭐ 强制硬限制：每场景最多3种方法论（1主+2辅）- 此约束不可违背

def match_methodology(scene_result: dict, transcript: str) -> dict:
    """
    根据场景识别结果，匹配最优方法论组合
    ⭐⭐⭐ 强制约束：总方法论数量 ≤ 3（1个主方法论 + 最多2个辅助方法论）
    ⚠️ 此约束为强制要求，任何情况下都不可违背

    返回: {
        "primary_methodology": {
            "name": "推拉力理论",
            "code": "M3",
            "reason": "离职访谈的核心分析框架",
            "analysis_dimensions": ["推力", "拉力", "留任力", "移动障碍"],
            "weight": 0.6
        },
        "secondary_methodologies": [...],  # 最多2个，强制限制
        "total_methodologies": 1 + len(secondary),  # 必须保证 ≤ 3
        "analysis_strategy": "...",
        "special_flags": ["挽留评估", "组织改进建议"]
    }
    """
    scene = scene_result["primary_scene"]
    confidence = scene_result["confidence"]
    sub_scenes = scene_result.get("secondary_scenes", [])
    
    # 基础映射
    primary = SCENE_METHOD_MAP[scene]["primary"]
    secondary = list(SCENE_METHOD_MAP[scene]["secondary"])  # 拷贝避免修改原映射
    
    # 动态调整：如果存在次要场景，融入其方法论
    for sub in sub_scenes:
        sub_methods = SCENE_METHOD_MAP[sub]["primary"]
        if sub_methods not in [primary] + secondary:
            secondary.append(sub_methods)
    
    # 内容驱动的微调
    # 例：绩效访谈中发现大量组织因素 → 增加杨三角分析
    content_signals = detect_content_signals(transcript)
    if "organizational_issues" in content_signals and "M1" not in [primary["code"]] + [s["code"] for s in secondary]:
        secondary.append(METHODOLOGIES["M1"])
    
    # ⭐⭐ 强制执行组织诊断方法论限制：杨三角和六盒模型最多只选一个
    # 优先使用杨三角理论(M1)，如果包含六盒模型(M2)则替换为杨三角
    org_diagnosis_codes = ["M1", "M2"]  # 组织诊断类方法论代码
    current_codes = [primary["code"]] + [s["code"] for s in secondary]
    org_count = len([c for c in current_codes if c in org_diagnosis_codes])
    
    if org_count > 1:
        # 如果同时有M1和M2，保留M1，移除M2
        if "M2" in current_codes:
            if secondary and secondary[0]["code"] == "M2":
                secondary = [s for s in secondary if s["code"] != "M2"]
            elif primary["code"] == "M2":
                primary = METHODOLOGIES["M1"]
    
    # ⭐ 强制执行方法论数量限制：辅助方法论最多2个
    MAX_SECONDARY = MAX_METHODOLOGIES - 1  # = 2
    if len(secondary) > MAX_SECONDARY:
        # 按优先级排序，保留优先级最高的2个辅助方法论
        secondary = sorted(secondary, key=lambda m: m.get("priority", 0), reverse=True)
        secondary = secondary[:MAX_SECONDARY]
    
    # 如果访谈涉及 AI 话题，激活 AI 增强维度（AI维度不计入方法论数量限制）
    ai_signals = detect_ai_signals(transcript)
    
    return {
        "primary_methodology": primary,
        "secondary_methodologies": secondary,  # 保证 ≤ 2个
        "total_methodologies": 1 + len(secondary),  # 保证 ≤ 3
        "ai_enhancement": ai_signals,
        "analysis_strategy": generate_strategy(scene, primary, secondary),
        "special_flags": detect_special_flags(transcript, scene)
    }
```

### M1: 杨三角理论 - 组织能力诊断分析

> **参考文档：** [yang-triangle-analysis.md](references/yang-triangle-analysis.md)

**适用场景：** 组织诊断、绩效（组织层面）、团队反馈

#### 分析维度

```
                    员工思维 (Mindset)
                   愿不愿意？
                      ▲
                     / \
                    / ⭐ \
                   /       \
                  /  组织    \
                 /   能力     \
                /             \
               ╱               ╲
              ╱                 ╲
员工能力 ◄───────────────────────► 员工治理
(Competence)                     (Governance)
 会不会？                          允不允许？
```

**三大支柱分析框架：**

| 支柱 | 核心问题 | 分析维度 | 评估标准 |
|------|---------|---------|---------|
| **员工思维** | 员工愿不愿意？ | ① 价值认同度 ② 文化理解度 ③ 战略认同度 ④ 变革接受度 | 🟢高度认同 / 🟡部分认同 / 🔴明显抵触 |
| **员工能力** | 员工会不会？ | ① 专业技能 ② 学习能力 ③ 协作能力 ④ 创新能力 ⑤ AI能力 | 🟢超出要求 / 🟡基本胜任 / 🔴明显不足 |
| **员工治理** | 组织允不允许？ | ① 流程效率 ② 资源配置 ③ 授权机制 ④ 激励制度 ⑤ 信息流通 | 🟢运作良好 / 🟡有改进空间 / 🔴严重阻碍 |

**分析步骤：**
1. **提取三支柱相关陈述**：将访谈内容按思维/能力/治理分类
2. **支柱健康度评估**：对每个支柱打分（1-5分）
3. **找到短板支柱**：识别最薄弱的支柱（木桶效应）
4. **根因分析**：对短板支柱使用5 Whys深挖
5. **交叉影响分析**：分析支柱间的相互影响
6. **行动建议**：基于短板生成优先改进建议

**杨三角归因模型（用于根因定位）：**

| 归因类型 | 问题信号 | 分析方向 |
|---------|---------|---------|
| **思维归因** | "不想做"、"不认同"、"没意义" | 价值观/文化/方向 |
| **能力归因** | "不会做"、"没能力"、"太难了" | 培训/招聘/能力建设 |
| **治理归因** | "做不了"、"没资源"、"不允许" | 流程/制度/资源/授权 |

### M2: 韦斯伯德六盒模型 - 系统性组织诊断分析

> **参考文档：** [weisbord-six-box-analysis.md](references/weisbord-six-box-analysis.md)

**适用场景：** 组织诊断（全面）、团队反馈、入职适应

#### 分析维度

```
┌─────────────────────────────────────────┐
│              外部环境                    │
├─────────────┬─────────────┬─────────────┤
│   ① 目的    │   ② 结构    │   ③ 关系    │
│  目标清晰？  │  分工合理？  │  协作顺畅？ │
├─────────────┼─────────────┼─────────────┤
│   ④ 奖励    │   ⑤ 领导    │  ⑥ 支持机制 │
│  激励有效？  │  方向明确？  │  工具到位？ │
└─────────────┴─────────────┴─────────────┘
```

**六维度分析框架：**

| 维度 | 分析焦点 | 系统性问题信号 | 区分个人抱怨 |
|------|---------|-------------|------------|
| **目的** | 目标清晰度、认同度、对齐度 | 目标传达机制缺失 | 仅个人不清楚（个人）vs 普遍不清楚（系统） |
| **结构** | 分工合理性、层级效率、跨部门协作 | 责任不清/流程冗余 | 个人能力不足 vs 岗位设计问题 |
| **关系** | 团队协作、部门间关系、冲突管理 | 系统性协作障碍 | 个人性格冲突 vs 结构性协作壁垒 |
| **奖励** | 薪酬公平、晋升机制、非物质激励 | 激励机制与目标错位 | 个人期望过高 vs 制度性不公 |
| **领导** | 方向清晰度、决策效率、管理赋能 | 领导力系统性缺失 | 个人好恶 vs 管理机制问题 |
| **支持机制** | 工具/系统/流程/AI基建支持 | 基础设施系统性不足 | 个人不会用 vs 工具缺失 |

**分析步骤：**
1. **内容分类**：将访谈陈述归类到6个维度
2. **问题分类**：区分「系统性问题」vs「个人抱怨」
3. **问题评分**：每个维度评分（影响范围 × 发生频率）
4. **维度排序**：按严重程度排序
5. **关联分析**：分析维度间的因果关系
6. **优先级矩阵**：生成问题优先级矩阵

**问题分类矩阵：**

```
           影响范围
        小           大
频  高 │ 快速修复  │ 🔴 紧急 │
率     │          │  优先   │
    ───┼──────────┼────────┤
    低 │ 观察监控  │ 规划   │
       │          │ 解决   │
       └──────────┴────────┘
```

### M3: 推拉力理论 - 离职决策分析

> **参考文档：** [push-pull-analysis.md](references/push-pull-analysis.md)

**适用场景：** 离职访谈、挽留访谈

#### 分析维度

```
┌─────────────────────────────────────────────────────┐
│                  离职决策力量分析                      │
├─────────────────────┬───────────────────────────────┤
│    推力 (Push) →    │    ← 拉力 (Pull)              │
│    现有工作负面因素  │    新机会正面因素              │
├─────────────────────┼───────────────────────────────┤
│    留任力 (Stay)    │    移动障碍 (Barrier)          │
│    留下的理由       │    离开的阻力                  │
└─────────────────────┴───────────────────────────────┘

离职决策公式：推力 > 留任力 + 移动障碍 - 拉力 → 离职
```

**四类力量分析框架：**

| 力量类型 | 分析维度 | 关键信号 |
|---------|---------|---------|
| **推力** | ① 工作不满 ② 发展受限 ③ 管理问题 ④ 薪酬不公 ⑤ 文化不适 ⑥ 工作负荷 ⑦ 技术栈过时 | 负面情绪表达、抱怨频率 |
| **拉力** | ① 更好薪酬 ② 更好发展 ③ 更好文化 ④ 更好平衡 ⑤ 新挑战 ⑥ 行业机会 | 对外部机会的描述、比较 |
| **留任力** | ① 情感承诺 ② 经济依赖 ③ 发展潜力 ④ 人际关系 ⑤ 项目牵挂 ⑥ 企业认同 | 积极情感、犹豫表达 |
| **移动障碍** | ① 转换成本 ② 家庭因素 ③ 地理限制 ④ 竞业限制 ⑤ 风险规避 ⑥ 市场环境 | 顾虑表达、不确定感 |

**挽留可能性评估矩阵：**

| 维度 | 🔴 低（难挽留） | 🟡 中（可挽留） | 🟢 高（易挽留） |
|------|-------------|-------------|-------------|
| **推力** | 多个强推力且长期积累 | 1-2个可解决的推力 | 推力弱或已有改善趋势 |
| **拉力** | 已拿到明确offer | 在看机会但未确定 | 未主动寻找外部机会 |
| **留任力** | 几乎无留任因素 | 有一定留任因素 | 多个强留任因素 |
| **移动障碍** | 障碍极低（年轻/无牵挂） | 有一定障碍 | 障碍较高（家庭/竞业） |

**综合挽留可能性评估：**
- 🔴 **低可能性**：推力强 + 拉力强 + 留任力弱 + 障碍低 → 建议做好交接
- 🟡 **中等可能性**：存在可解决的核心推力 → 制定针对性挽留方案
- 🟢 **高可能性**：推力弱或可解决 + 留任力强 → 及时干预可挽回

### M4: 冰山模型 - 胜任力评估分析

> **参考文档：** [iceberg-analysis.md](references/iceberg-analysis.md)

**适用场景：** 招聘面试、晋升评估、转岗评估

#### 分析维度

```
                    ┌─────────────────┐
                    │   水面以上      │
                    │   (可见 20%)    │
                    ├─────────────────┤
                    │ ① 知识          │  ← 容易评估和培养
                    │ ② 技能          │
                    ├═════════════════┤ ← 水面线 ═══════
                    │ ③ 社会角色      │
        水面以下    │ ④ 自我认知      │  ← 难以改变
        (隐藏 80%)  │ ⑤ 特质          │    决定长期绩效
                    │ ⑥ 动机          │
                    └─────────────────┘
```

**六层分析框架：**

| 层次 | 分析焦点 | 评估方式 | 培养难度 |
|------|---------|---------|---------|
| **知识** | 专业知识储备 | 直接提问、经验描述 | ⭐ 容易培养 |
| **技能** | 实际操作能力 | 案例分析、过往成果 | ⭐⭐ 可培养 |
| **社会角色** | 角色定位和职业认知 | 行为模式分析 | ⭐⭐⭐ 较难改变 |
| **自我认知** | 对自身的客观评估 | 自我评价vs他人评价 | ⭐⭐⭐⭐ 难改变 |
| **特质** | 人格特质和行为倾向 | 行为一致性分析 | ⭐⭐⭐⭐ 难改变 |
| **动机** | 内在驱动力 | 选择模式和投入度分析 | ⭐⭐⭐⭐⭐ 极难改变 |

**分析步骤：**
1. **逐层提取**：从访谈记录中提取六层证据
2. **证据质量评估**：区分表面陈述和深层表露
3. **岗位匹配分析**：与目标岗位要求对比
4. **水面以下重点**：重点分析③④⑤⑥层
5. **风险评估**：识别「水面以下」的红旗信号
6. **发展建议**：基于可培养度制定发展计划

### M5: 吉尔伯特 BEM 模型 - 绩效差距分析

> **参考文档：** [gilbert-bem-analysis.md](references/gilbert-bem-analysis.md)

**适用场景：** 绩效访谈、入职适应、员工发展

#### 分析维度

```
┌─────────────────────────────────────────────────────┐
│              影响绩效的六大因素分析                    │
├─────────────────────┬───────────────────────────────┤
│  环境因素 (75%) ⭐   │    个人因素 (25%)             │
│  先分析环境！        │    后分析个人！               │
├─────────────────────┼───────────────────────────────┤
│  ① 信息与反馈       │   ④ 知识与技能                │
│  目标清晰？反馈及时？│   具备所需知识技能？          │
├─────────────────────┼───────────────────────────────┤
│  ② 资源与工具       │   ⑤ 天赋与能力                │
│  工具到位？资源充足？│   天赋匹配？潜力如何？        │
├─────────────────────┼───────────────────────────────┤
│  ③ 激励与后果       │   ⑥ 动机与态度                │
│  激励有效？后果明确？│   有内在动机？态度积极？      │
└─────────────────────┴───────────────────────────────┘
```

**核心分析原则：「先环境后个人」**

> 绩效问题的 75% 可归因于环境因素。分析时**必须先排查环境因素**，避免「培训万能论」——不要在环境问题未解决时就归咎于个人能力不足。

**六因素分析框架：**

| 因素 | 核心问题 | 分析方向 | 改进类型 |
|------|---------|---------|---------|
| **信息与反馈** | 员工知道该做什么吗？ | 目标传达 → 反馈频次 → 信息质量 | 管理改进 |
| **资源与工具** | 员工有条件做好吗？ | 工具支持 → 时间充裕 → 预算充足 | 资源投入 |
| **激励与后果** | 做好了有什么好处？ | 薪酬激励 → 晋升通道 → 认可机制 | 制度优化 |
| **知识与技能** | 员工有能力做吗？ | 专业知识 → 技术技能 → 经验积累 | 培训发展 |
| **天赋与能力** | 员工适合做这个吗？ | 逻辑思维 → 沟通能力 → 学习速度 | 岗位适配 |
| **动机与态度** | 员工想做好吗？ | 内在兴趣 → 职业承诺 → 积极主动 | 激励策略 |

**绩效差距根因定位流程：**
1. **明确绩效差距**：期望绩效 vs 实际绩效 → 差距描述
2. **环境因素扫描**（先）：逐一分析信息/资源/激励三因素
3. **个人因素扫描**（后）：逐一分析知识/天赋/动机三因素
4. **根因排序**：按影响程度排序主要根因
5. **改进方案**：针对根因制定短/中/长期改进计划

### M6: 职业锚理论 + GROW 模型 - 职业发展分析

> **参考文档：** [career-anchor-grow-analysis.md](references/career-anchor-grow-analysis.md)

**适用场景：** 职业发展、挽留访谈、转岗评估

#### 职业锚识别分析

**8 种职业锚类型识别框架：**

| 职业锚类型 | 核心特征 | 识别信号（从访谈中提取） |
|-----------|---------|----------------------|
| **技术/职能型** | 追求专业精进 | 强调技术深度、不愿做管理、对技术问题兴奋 |
| **管理型** | 追求管理责任 | 关注团队规模、喜欢决策、追求影响力 |
| **自主/独立型** | 追求工作自由 | 不喜欢约束、追求灵活性、抵触流程 |
| **安全/稳定型** | 追求稳定可预测 | 关注稳定性、厌恶变化、重视安全感 |
| **创业/创造型** | 追求创造新事物 | 喜欢从0到1、不满足于守成、创业冲动 |
| **服务/奉献型** | 追求帮助他人 | 强调社会价值、关心团队成长、公益导向 |
| **纯挑战型** | 追求战胜难题 | 喜欢高难度、竞争导向、追求极限 |
| **生活方式型** | 追求工作生活平衡 | 关注WLB、抵触加班、重视家庭时间 |

**职业锚识别方法（从访谈中提取线索）：**
1. **选择模式分析**：过往职业选择的核心考量
2. **满意度来源**：什么时候最有成就感
3. **不可放弃要素**：无论如何不愿放弃什么
4. **拒绝模式**：曾拒绝什么机会，为什么

#### GROW 模型分析框架

| 阶段 | 分析内容 | 评估维度 |
|------|---------|---------|
| **Goal（目标）** | 员工的职业目标是否清晰？ | 目标具体性 / 目标可行性 / 与组织对齐度 |
| **Reality（现实）** | 员工对自身现状的认知是否客观？ | 自我认知准确性 / 差距认知 / 资源盘点 |
| **Options（选择）** | 员工看到了哪些发展路径？ | 选项多样性 / 创造性 / 可行性评估 |
| **Will（意愿）** | 员工的行动承诺度如何？ | 承诺具体性 / 时间规划 / 障碍应对 |

### M7: STAR 面试法 - 行为评估分析

> **参考文档：** [star-analysis.md](references/star-analysis.md)

**适用场景：** 招聘面试、晋升评估

#### 分析维度

```
S (Situation) → T (Task) → A (Action) → R (Result)
   情境              任务        行动          结果
   ↓                 ↓          ↓             ↓
   背景是否清晰？    角色是否明确？ 行动是否具体？ 结果是否量化？
```

**STAR 评估框架：**

| 要素 | 评估标准 | 🟢 完整 | 🟡 部分 | 🔴 缺失 |
|------|---------|---------|---------|---------|
| **S-情境** | 背景清晰、有细节 | 详细描述具体场景 | 笼统描述 | 无法描述具体情境 |
| **T-任务** | 角色明确、目标清晰 | 清楚说明个人角色 | 模糊提及 | 混淆团队与个人贡献 |
| **A-行动** | 具体行动、有方法论 | 详述具体步骤 | 概括性描述 | 泛泛而谈/编造感 |
| **R-结果** | 量化成果、有影响 | 有数据支撑的成果 | 定性描述 | 无法说明具体结果 |

**STAR 完整度评分（每个行为事例）：**
- **4分**：S+T+A+R 全部完整且相互逻辑一致
- **3分**：缺少一个要素但其余完整
- **2分**：只有表面描述，缺乏细节
- **1分**：多个要素缺失，逻辑不连贯
- **0分**：无法提供有效行为事例

**红旗信号检测：**
| 红旗 | 表现 | 风险 |
|------|------|------|
| 🚩 过度使用"我们" | 无法区分个人贡献 | 可能不是核心参与者 |
| 🚩 结果含糊 | 说"效果很好"但无数据 | 可能结果不佳 |
| 🚩 逻辑断裂 | 行动与结果之间无因果 | 可能编造经历 |
| 🚩 角色膨胀 | 声称主导但细节矛盾 | 夸大贡献 |
| 🚩 时间跳跃 | 时间线混乱或前后矛盾 | 可能合并/编造经历 |

## 🔬 分析方法论：假设驱动 + 矛盾检测 + 深度追因

### 假设驱动分析法

每次分析开始前，基于场景和方法论**先形成假设**，然后用访谈内容验证/推翻假设：

```
Step 1: 基于场景生成初始假设
   例（离职分析）:
   H1: 薪酬不满是主要推力
   H2: 职业发展受限是次要推力
   H3: 直属上级管理问题可能是隐性推力

Step 2: 用访谈证据验证每个假设
   H1: 🟢 支持（多次提及薪酬对比）
   H2: 🟡 部分支持（提到发展但情绪不强烈）
   H3: 🔴 推翻（对上级评价积极）

Step 3: 修正假设，形成新假设
   H4: 工作挑战不足可能是核心推力（新发现）

Step 4: 二次验证，形成最终结论
```

### 矛盾检测机制

自动检测访谈中的**自相矛盾**和**表里不一**：

| 矛盾类型 | 表现 | 分析策略 |
|---------|------|---------|
| **言行矛盾** | 说满意但语气消极 | 更信任情感信号 |
| **前后矛盾** | 前面说好后面说差 | 关注后半段（防御降低后更真实） |
| **程度矛盾** | 说"还好"但举了很多反例 | 关注事例而非总结性评价 |
| **归因矛盾** | 归因于外部但行为模式显示内因 | 综合分析行为证据 |
| **社会期望偏差** | 说"公司很好"但在找工作 | 识别社会期望回答 |

### 5 Whys 深度追因

对关键发现使用 5 Whys 追根溯源：

```
发现: 员工绩效下滑
  Why 1: 为什么绩效下滑？→ 项目交付质量下降
  Why 2: 为什么质量下降？→ 加班频繁导致粗心
  Why 3: 为什么加班频繁？→ 人手不足+需求膨胀
  Why 4: 为什么人手不足？→ 预算冻结+离职未补
  Why 5: 为什么预算冻结？→ 组织战略调整中
  
  根因：组织战略调整期的人力资源配置问题（治理归因）
  → 非个人能力问题（BEM模型验证：环境因素主导）
```

## 📊 数据可视化分析（v2.2 增强版 - 自动判断）

> **v2.2.0 变更**：根据访谈内容自动判断是否需要生成数据可视化，无需询问用户

### 自动判断规则

**何时生成可视化：**

| 条件 | 说明 | 触发可视化 |
|------|------|-----------|
| **多维度评分** | 访谈分析中有3个以上维度的量化评分 | ✅ 自动生成 |
| **对比分析** | 存在多个对象、多个时间点、多个维度的对比 | ✅ 自动生成 |
| **趋势分析** | 多次访谈或时间序列数据 | ✅ 自动生成 |
| **优先级评估** | 问题评分、改进优先级等 | ✅ 自动生成 |
| **简单访谈** | 单次访谈、无量化评分、无对比需求 | ❌ 不生成 |

**判断逻辑：**
```python
def should_generate_visualization(analysis_result: dict) -> dict:
    """
    判断是否需要生成数据可视化
    返回: {
        "should_generate": True/False,
        "chart_types": ["radar", "bar", "line"],
        "reason": "判断理由"
    }
    """
    chart_types = []
    
    # 1. 检查是否有量化评分（3个以上维度）
    if len(analysis_result.get("scores", {})) >= 3:
        chart_types.append("radar")
    
    # 2. 检查是否有对比数据
    if analysis_result.get("has_comparison"):
        chart_types.append("bar")
    
    # 3. 检查是否有时间序列
    if analysis_result.get("has_trend"):
        chart_types.append("line")
    
    # 4. 决定是否生成
    should_generate = len(chart_types) > 0
    
    return {
        "should_generate": should_generate,
        "chart_types": chart_types,
        "reason": "检测到{}类适合可视化的数据".format(len(chart_types)) if should_generate else "访谈内容无需可视化"
    }
```

## 📚 方法论索引

本技能基于多种专业方法论，针对不同分析场景提供定制化分析框架：

### 🏛️ 组织诊断类

| 方法论 | 适用场景 | 参考文档 |
|-------|---------|---------|
| **杨三角理论** ⭐ | 组织健康度评估、快速诊断 | [yang-triangle-analysis.md](references/yang-triangle-analysis.md) |
| **用途** | 从员工思维、员工能力、员工治理三个支柱诊断组织能力 |
| **核心价值** | 简洁聚焦，源自中国企业实践，互联网公司优先使用 |

| **韦斯伯德六盒模型** | 组织全面诊断、系统性评估 | [weisbord-six-box-analysis.md](references/weisbord-six-box-analysis.md) |
| **用途** | 从目的、结构、关系、奖励、领导、支持机制六个维度诊断 |
| **核心价值** | 区分"系统性问题"与"个人抱怨"，更适合全面诊断 |

### 🚪 离职分析类

| 方法论 | 适用场景 | 参考文档 |
|-------|---------|---------|
| **推拉力理论** | 离职原因分析、挽留评估 | [push-pull-analysis.md](references/push-pull-analysis.md) |
| **用途** | 从推力、拉力、留任力、移动障碍四维度分析离职决策 |
| **核心价值** | 系统评估挽留可能性，量化分析离职驱动力 |

### 👥 人才评估类

| 方法论 | 适用场景 | 参考文档 |
|-------|---------|---------|
| **冰山模型** | 候选人评估、晋升评估、岗位适配 | [iceberg-analysis.md](references/iceberg-analysis.md) |
| **用途** | 从知识/技能/角色/认知/特质/动机六层评估人才 |
| **核心价值** | 深入评估"水面以下"80%的隐藏素质 |

| **STAR 面试法** | 面试评估、行为分析 | [star-analysis.md](references/star-analysis.md) |
| **用途** | 通过情境/任务/行动/结果四要素评估行为证据 |
| **核心价值** | 基于行为证据的客观评估，识别虚假/夸大 |

### 📈 绩效/发展类

| 方法论 | 适用场景 | 参考文档 |
|-------|---------|---------|
| **吉尔伯特 BEM 模型** | 绩效差距分析、发展计划 | [gilbert-bem-analysis.md](references/gilbert-bem-analysis.md) |
| **用途** | 从环境(75%)/个人(25%)因素分析绩效差距根因 |
| **核心价值** | 先环境后个人，避免"培训万能论" |

### 🎯 职业发展类

| 方法论 | 适用场景 | 参考文档 |
|-------|---------|---------|
| **职业锚理论 + GROW 模型** | 职业规划、发展分析 | [career-anchor-grow-analysis.md](references/career-anchor-grow-analysis.md) |
| **用途** | 识别员工职业锚，分析 Goal/Reality/Options/Will |
| **核心价值** | 基于员工真实动机分析发展方向 |
